RoleMesh · Kubernetes Deployment Network Topology

Rancher RKE2 · single namespace rolemesh · declarative infrastructure (Helm) + verify-only startup · design rev4
Platform traffic (orchestrator / webui) Agent traffic (only two destinations) Gateway traffic (allowlisted egress / NATS RPC) Denied by NetworkPolicy Control-plane action

① agent-default-deny

Selector role=agent; all ingress + egress denied, zero rules. The equivalent of Docker's Internal=true bridge — kept as its own policy so startup verification can check the "default deny" invariant in isolation.

② agent-allow-egress

Allows only agent → gateway (53 / 3001 / 3128) and → NATS (4222). Cluster DNS is not allowed: name resolution goes only through the gateway's allowlist resolver (DNS-exfiltration control).

③ gateway-policy

Ingress only from agents / orchestrator; egress allows NATS (4222, required for credential RPC), 53/80/443 plus ports declared in values. An app-level domain allowlist sits on top — defense in depth.

④ orchestrator-policy

Egress only → K8s API, Postgres, NATS, gateway, and external HTTPS (its own LLM calls). RBAC is limited to pods create/delete/watch/log within this namespace.

Identity model: the orchestrator mints an HMAC token at spawn time and injects it into the agent; the token travels in-band with every request (reverse-proxy path segment / Proxy-Authorization) and the gateway verifies it statelessly against a shared secret — independent of network topology, identical across the Docker and K8s runtimes.  |  Source: docs/container-runtime-decoupling-design.md (rev4, 2026-06-12)
