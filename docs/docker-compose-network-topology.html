RoleMesh · Local Docker / Compose Network Topology

Docker host (Ubuntu amd64) · docker compose · declarative networks + verify-only startup · design rev4
Platform traffic (orchestrator / webui) Agent traffic (only two destinations) Gateway traffic (egress / NATS RPC) Denied / docker control Spawn (docker.sock)

① agent-net (internal: true)

A bridge with no default route — agents simply have no path to the internet. This is the Docker equivalent of the K8s default-deny egress NetworkPolicy. Declared in compose; startup verifies the Internal flag rather than creating the network.

② DNS pinned to the gateway

HostConfig.Dns = [172.28.100.53] forces every agent's resolver to the gateway (listening on :53, cap NET_BIND_SERVICE). Allowlisted names recurse to Docker's embedded DNS (127.0.0.11); everything else → NXDOMAIN.

③ Spawn via docker.sock (DooD)

The orchestrator (itself a container) drives the host dockerd over the bound socket to create/destroy agent containers — the only imperative container op left. Bind mounts use ROLEMESH_HOST_DATA_DIR for host-path translation.

④ Dual-homed gateway

One container on both agent-net (fixed IP = EGRESS_GATEWAY_DNS_IP, static config) and egress-net (the normal bridge out). The sole path from the isolated bridge to the internet — same token / credential model as production.

Identity model: the orchestrator mints an HMAC token at spawn time and injects it into the agent; the token travels in-band with every request (reverse-proxy path segment / Proxy-Authorization) and the gateway verifies it statelessly against a shared secret — identical to the K8s runtime.  |  Same guarantees as the Kubernetes deployment, realized with Docker primitives (internal bridge / HostConfig.Dns / docker.sock) instead of NetworkPolicy / dnsConfig / the K8s API.  |  Source: docs/container-runtime-decoupling-design.md (rev4, 2026-06-12)
